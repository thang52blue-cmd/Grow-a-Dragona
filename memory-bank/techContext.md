## Known gaps

- `ci/compile-check.sh` now passes `--definitions="@roblox=ci/luau-lsp/globalTypes.d.luau"` to
  `luau-lsp analyze`, so `game`, `script`, and other engine globals resolve to real Roblox classes
  instead of erroring as unknown. That file was downloaded from
  `JohnnyMorganz/luau-lsp`'s `scripts/globalTypes.d.luau` at the pinned `1.68.1` tag (matches the
  `luau-lsp` version in `rokit.toml`) and is committed so the gate works offline/deterministically.
  Re-download it if `luau-lsp` is ever re-pinned to a different version.
- **Verified gap, not assumed:** with those definitions loaded, `luau-lsp analyze` correctly flags a
  plain type mismatch (`local x: number = "a string"` errors as expected), but calling a
  nonexistent/mistyped method on an `Instance`-derived value (e.g. `dataStore:TypoedMethod()` on a
  `DataStore` obtained via `DataStoreService:GetDataStore(...)`) does **not** get flagged, even
  though the `DataStore` class and its real members are present in the definitions file. Root cause
  not yet isolated (tried `--no-strict-dm-types` toggling; behavior didn't change). Practical effect:
  `ci/compile-check.sh` reliably catches ordinary Luau type errors, but is not a strong guarantee
  against Instance/API misuse in `src/server/`/`src/client/` â€” don't rely on a green compile-check
  alone for that class of bug; a human/Studio pass still matters there.
- `selene.toml` (repo root, `std = "roblox"`) and `roblox.yml` (generated by
  `selene generate-roblox-std`, committed at repo root, ~32k lines) are both required for
  `ci/lint.sh`'s selene pass to work at all. **Verified by hitting it for real:** without
  `selene.toml`, selene defaults to a non-Luau grammar and its parser (`full_moon`) crashes
  (`internal error: entered unreachable code`) on ordinary Luau syntax like `::` type ascription,
  `function f(): ReturnType`, and `export type`. Regenerate `roblox.yml` if `selene` is ever
  re-pinned (`selene generate-roblox-std` from the repo root).
- `src/shared/Domain/SessionLock.luau` only holds the pure claim/reject *decision* (given a session
  record, jobId, now, timeout). The actual `os.time()`/`game.JobId` call sites live in
  `src/server/Persistence/DataService.luau`, which can't run under Lune â€” this split is why the
  session-lock rule is Lune-testable at all despite needing engine-only data to invoke for real.

## Commands

```bash
rokit install                # one-time per machine
ci/gate-freshness.sh          # FRESH/STALE check against the last stamped green run
ci/gate-freshness.sh --stamp  # record current commit+dirty-state as green (run after a real green)
ci/compile-check.sh           # rojo build + luau-lsp analyze -> COMPILE_OK | COMPILE_ERROR
ci/run-tests.sh fast          # lune-run every src/**/*.spec.luau -> PASSED | TESTS_FAILED | NO_TESTS
ci/run-tests.sh engine        # deferred until the activation ADR (backlog #3) -> NO_TESTS for now
ci/lint.sh                    # selene + stylua --check, advisory -> never blocks
rojo serve                    # local Rojo server for Studio to connect to
```

All scripts print a single machine-readable JSON line last; that `status` field is ground truth, not
the shell exit code (`ci/lint.sh` exits 0 even when it reports issues, since it's advisory).

## Test-file convention

A `*.spec.luau` file is a Lune script: it must exit non-zero (e.g. via `process.exit(1)` after
reporting failures) if any assertion inside it fails, and exit 0 otherwise. `ci/run-tests.sh` treats
each spec file as pass/fail purely by its process exit code â€” no shared test-framework dependency is
assumed yet. Revisit this if/when a real spec-running library is adopted (needs an ADR per AGENTS.md
"no Promise library / dependency changes without one" spirit â€” same applies to test frameworks going
into `wally.toml`).
