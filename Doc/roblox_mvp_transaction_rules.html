Transaction Rules Tối Thiểu

Bản kiến trúc rút gọn cho prototype Roblox. Chỉ giữ các phần cần thiết để core loop chạy ổn định, không mất dữ liệu, không duplicate item và không để client tự tạo Gold, Egg, Dragon hoặc reward.

Server Authority + Player Queue + Atomic Commit + Safe Save

1. Mục tiêu của phase MVP

Core loop chạy được

Buy Egg → Hatch → Feed → Adult → Produce → Collect → Sell.

Không lỗi economy cơ bản

Không âm inventory, không double claim, không dùng một item hai lần.

Dễ mở rộng

Không xây framework quá lớn nhưng vẫn có boundary rõ để thêm Display và Raid sau.

MVP không cần chống mọi kiểu exploit nâng cao. Mục tiêu là loại bỏ những lỗ hổng nghiêm trọng nhất ngay từ data layer.

2. Các rule tối thiểu phải giữ

1
Client chỉ gửi intentClient không gửi final price, reward, odds hoặc timer.
2
Server là source of truthOwnership, currency, inventory và job state lấy từ server profile.
3
Mọi persistent mutation đi qua transactionKhông service nào tự sửa profile ngoài TransactionService.
4
Mỗi player chỉ chạy một transaction tại một thời điểmDùng per-player queue.
5
Validate không mutateChỉ kiểm tra request và current state.
6
Stage không mutateTính NewGold, NewAmount, NewJob trong local patch.
7
Commit không yieldKhông task.wait, DataStore hoặc InvokeClient trong commit.
8
Sau commit không rollback gameplayMark dirty và save lại; không refund theo chuỗi thao tác ngược.
9
Hatch result roll ở serverClient chỉ hiển thị animation.
10
Runtime Part không phải dữ liệu thậtInventory/Nest/HatchJob trong profile mới là source of truth.

3. Flow transaction MVP

Client sends intent │ ▼ RemoteGateway ├── Validate payload type/range ├── Check profile loaded ├── Basic rate limit └── Route action │ ▼ PerPlayerTransactionQueue │ ▼ Transaction Handler ├── Validate ownership/state ├── Stage local patch └── Commit Atomic Write Set │ ▼ DataService ├── Revision += 1 ├── MarkDirty ├── PrioritySave if important └── Autosave / leave save
Đây là flow đủ dùng cho MVP. Chưa cần distributed lock, saga hoặc hệ thống anti-cheat phức tạp.

5. Các transaction cần có trong MVP

Ưu tiênTransactionLý do
1BuyEggTransactionCore economy đầu vào.
2StartHatchTransactionConsume egg và tạo persistent job.
3ClaimHatchTransactionTạo Dragon và chống double claim.
4FeedDragonTransactionConsume food và tăng growth.
5CollectNestTransactionResolve production và chuyển egg vào inventory.
6SellProductionEggTransactionCore economy đầu ra.
Display, Raid, Trade và Marketplace chưa cần nằm trong MVP transaction set nếu core loop chưa ổn định.

6. Atomic Write Set tối thiểu

BuyEggTransaction

Gold -= TotalPrice HatchEgg += Amount Statistics.TotalGoldSpent += TotalPrice Revision += 1

StartHatchTransaction

HatchEgg -= 1 Hatching.Slot.Job = NewHatchJob NextIds.HatchJob += 1 Revision += 1

ClaimHatchTransaction

Dragons += NewDragon Hatching.Slot.Job = nil NextIds.Dragon += 1 Revision += 1

FeedDragonTransaction

Food -= Amount Dragon.GrowthXP = NewGrowthXP if JustReachedAdult then Dragon.ReachedAdultAt = TransactionTime Production.Producers += NewProducer end Revision += 1

CollectNestTransaction

Resolve Production to TransactionTime Nest.Uncollected = empty Inventory.ProductionEggs += CollectedEggs Revision += 1

SellProductionEggTransaction

ProductionEgg -= Amount Gold += ServerCalculatedReward Revision += 1
Nếu một action cần thay đổi nhiều system để giữ dữ liệu hợp lệ, các thay đổi đó phải nằm trong cùng commit.

7. Persistence tối thiểu

Một profile key cho mỗi player

Giữ currency, inventory, dragons, jobs và production trong cùng một profile object.

Một server sở hữu profile

Không cho hai server cùng mutate profile một lúc.

ActionSave mode
Feed DragonMark dirty, autosave.
Collect NestMark dirty, autosave.
Buy EggMark dirty hoặc priority save.
Start HatchPriority save.
Claim HatchPriority save.
Player leaveSave latest committed revision.
Transaction commit → Revision += 1 → MarkDirty → SaveScheduler coalesces changes → Save latest profile → Never save older revision over newer revision
Không gọi DataStore save sau mỗi lần Feed. Cần autosave và gộp nhiều thay đổi để tránh throttling.

9. Các phần để phase sau

Chưa cần triển khai trong MVP

  • Trade transaction giữa hai player.
  • Marketplace global.
  • Distributed locks qua MemoryStore.
  • Guild inventory.
  • Persistent receipt cho mọi action nhỏ.
  • Suspicion score và auto-ban phức tạp.
  • Parallel Luau hoặc Actor-based transaction processing.
  • Durable save cho mọi transaction.
  • Generic transaction DSL.
  • Cross-server raid state.
Chỉ thêm các phần trên khi feature thực sự tồn tại. Không xây trước một hệ thống phân tán cho core loop single-profile.

12. MVP Definition of Done

☐ Client chỉ gửi intent

☐ Có PayloadValidator
☐ Có RateLimiter và queue cap
☐ Mỗi player có transaction queue
☐ Validate và Stage không mutate profile
☐ Commit không yield
☐ Atomic Write Set đầy đủ
☐ Mỗi commit tăng Revision

☐ Không double claim
